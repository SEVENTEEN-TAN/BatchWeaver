1. 核心设计思路
我们将系统划分为两个完全隔离的域：

管理域 (Management Domain)：负责 Spring Batch 框架运行所需的元数据（JobInstance, StepExecution 等）。使用独立的数据源和事务管理器，确保即使业务逻辑报错，运行记录也能成功持久化。

业务域 (Business Domain)：负责实际的数据逻辑。利用 MyBatis-Flex 的流式查询和批量写入能力，通过业务事务管理器控制 Step 内的数据一致性。

2. 详细实现方案
第一步：双数据源与事务管理器配置
在 Spring Boot 3.x 中，通过 @BatchDataSource 注解可以优雅地指定元数据存储位置。

Java

@Configuration
public class DataSourceConfig {

    // --- 1. 业务数据源与事务管理器 ---
    @Bean
    @Primary
    @ConfigurationProperties("spring.datasource.business")
    public DataSource businessDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    @Primary
    public PlatformTransactionManager businessTransactionManager(DataSource businessDataSource) {
        // 绑定业务数据源，用于 Step 中的 Chunk 事务
        return new DataSourceTransactionManager(businessDataSource);
    }

    // --- 2. Batch 元数据数据源与事务管理器 ---
    @Bean
    @BatchDataSource // 关键：告知 Spring Batch 使用此数据源存储元数据
    @ConfigurationProperties("spring.datasource.batch")
    public DataSource batchDataSource() {
        return DataSourceBuilder.create().build();
    }

    @Bean
    public PlatformTransactionManager batchTransactionManager(@BatchDataSource DataSource ds) {
        // 用于管理 JobRepository 的状态更新
        return new DataSourceTransactionManager(ds);
    }
}
第二步：MyBatis-Flex 核心配置
确保 MyBatis-Flex 绑定到 businessDataSource。

Java

@Configuration
public class MybatisFlexConfig {

    @Bean
    public SqlSessionFactory sqlSessionFactory(DataSource businessDataSource) throws Exception {
        SqlSessionFactoryBean factoryBean = new SqlSessionFactoryBean();
        factoryBean.setDataSource(businessDataSource);
        
        // 注入 MyBatis-Flex 的配置
        FlexConfiguration configuration = new FlexConfiguration();
        configuration.setLogImpl(StdOutImpl.class); // 可选：日志配置
        factoryBean.setConfiguration(configuration);
        
        return factoryBean.getObject();
    }
}
第三步：高性能 ItemReader 实现 (流式读取)
为了处理百万级数据，我们不建议使用分页查询，而是利用 MyBatis-Flex 的流式查询。

Java

@Component
@StepScope // 允许使用 JobParameters
public class FlexCursorReader extends AbstractItemCountingItemStreamItemReader<User> {

    private final UserMapper userMapper;
    private Cursor<User> cursor;
    private Iterator<User> iterator;

    public FlexCursorReader(UserMapper userMapper) {
        this.userMapper = userMapper;
        setName("flexCursorReader");
    }

    @Override
    protected void doOpen() throws Exception {
        // 使用 MyBatis-Flex 的 QueryWrapper 构建流式查询
        QueryWrapper queryWrapper = QueryWrapper.create()
                .select(USER.ALL_COLUMNS)
                .from(USER)
                .where(USER.STATUS.eq(1));
        
        // 这里的执行会保持数据库连接，直到 Step 结束
        this.cursor = userMapper.selectCursorByQuery(queryWrapper);
        this.iterator = cursor.iterator();
    }

    @Override
    protected User doRead() {
        return (iterator != null && iterator.hasNext()) ? iterator.next() : null;
    }

    @Override
    protected void doClose() throws Exception {
        if (cursor != null) cursor.close();
    }
}
第四步：高性能 ItemWriter 实现 (批量写入)
利用 MyBatis-Flex 的 Db.executeBatch 结合 Spring Batch 的 Chunk 机制。

Java

@Component
public class FlexBatchWriter implements ItemWriter<UserEntity> {

    @Override
    public void write(Chunk<? extends UserEntity> chunk) throws Exception {
        // chunk.getItems() 获取的是当前批次的数据
        Db.executeBatch(chunk.getItems(), (mapper, entity) -> {
            mapper.insert(entity);
        }, UserEntityMapper.class);
        
        // 注意：无需手动 commit，Spring Batch 会在 write 执行完后
        // 通过 businessTransactionManager 自动提交当前 Chunk
    }
}
第五步：Job 与 Step 的编排
Java

@Configuration
public class MyJobConfig {

    @Bean
    public Job syncUserJob(JobRepository jobRepository, Step syncStep) {
        return new JobBuilder("syncUserJob", jobRepository)
                .start(syncStep)
                .build();
    }

    @Bean
    public Step syncStep(JobRepository jobRepository,
                        @Qualifier("businessTransactionManager") PlatformTransactionManager businessTxManager,
                        FlexCursorReader reader,
                        FlexBatchWriter writer) {
        return new StepBuilder("syncStep", jobRepository)
                .<User, UserEntity>chunk(1000, businessTxManager) // 关键：注入业务事务
                .reader(reader)
                .processor(item -> {
                    // 业务转换逻辑
                    return new UserEntity(item.getName());
                })
                .writer(writer)
                .faultTolerant()
                .retryLimit(3)
                .retry(DeadlockLoserDataAccessException.class) // 重试策略
                .build();
    }
}
3. 该方案的优势分析
1. 事务行为深度解析
成功场景：Step 每处理 1000 条（Chunk Size），businessTransactionManager 提交一次业务数据；同时 batchTransactionManager 另开事务更新 BATCH_STEP_EXECUTION 表。

失败场景：如果在第 1500 条报错，businessTransactionManager 回滚当前 Chunk（即回滚第 1001-1500 条），但 batchTransactionManager 会不受影响地将 Step 状态标记为 FAILED。

2. 性能保障
内存安全：selectCursorByQuery 保证了即使源表有千万数据，Java 内存也只持有当前处理的一小部分。

IO 效率：Db.executeBatch 将原本 1000 次的插入请求，在数据库驱动层合并为更少的网络交互。

3. 可观测性
由于元数据独立存储，即使你的业务数据库因为压力过大导致连接池爆满或死锁，Spring Batch 依然能将“由于连接超时导致失败”这一结果写入元数据库，方便你后续排查。
